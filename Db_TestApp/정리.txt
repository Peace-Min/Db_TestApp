1번 테스트에 대해 어떤 코멘트로 소개할지 추천좀 현재는 1번 기능에 대해 DB write 트랜잭션만 수행하는 Thread를 생성하여 DB 모드에 따른 성능비교를 테스트한다.

총 레코드 수 : 성능 비교를 위한 DB에 Write 총 데이터 개수
트랜잭션 당 레코드 수 : 트랜잭션 수행당 DB에 write할 개수(트랜잭션 횟수랑 연관됨을 강조하고싶음)

2번 테스트는 어떤 코멘트로 소개 그리고 1단계 2단계 어떻게 동작하여 기능을 수행하는지 설명
지금은
1. DB Write 트랜잭션만 수행하는 Thread를 생성하여 Write를 수행한다.
2. Db Write Thread와 DB Read Thread를 동시에 생성후 Write,Read를 동시에 수행한다.
=> 동시성 테스트 결과는 단순 CPU 점유에 의한 성능차이 발생으로 확인 이렇게 결론냈는데 수정좀해줘 깔끔하게

/////////////////////
1. 단일 쓰기 스레드에서 WAL / MEMORY 모드별로 동일한 양의 데이터를 Write 트랜잭션 성능을 비교합니다.

총 레코드 수 : 성능 비교를 위한 DB에 Write 총 데이터 개수

트랜잭션 당 레코드 수 : 한 번의 트랜잭션(Commit) 안에 묶어서 Write 할 데이터의 개수
실제 트랜잭션 수행 횟수 = 총 레코드 수 / 트랜잭션 당 레코드 수
=> 값이 작을수록 트랜잭션을 자주 수행하고, 값이 클수록 트랜잭션을 적게 수행합니다.

/////////////////////
1. 단일 쓰기 스레드에서 WAL / MEMORY 두 모드에 대해 동일한 양의 데이터를 기록할 때의 쓰기 트랜잭션 성능을 비교합니다.

총 레코드 수: 성능 비교를 위해 DB에 Write할 전체 데이터(행)의 개수입니다.

트랜잭션 당 레코드 수: 한 번의 트랜잭션안에 묶어서 Write할 데이터의 개수입니다.
 - 실제 트랜잭션 수행 횟수 = 총 레코드 수 / 트랜잭션 당 레코드 수
 → 값이 작을수록 트랜잭션을 더 자주 수행하고, 값이 클수록 트랜잭션을 더 적게 수행합니다.


2. WAL 모드에서 Writer와 Reader를 동시에 동작시켰을 때, Reader 부하가 Write 트랜잭션 성능에 얼마나 영향을 주는지 확인하는 동시성 테스트입니다.
1. DB Write 전용 스레드 하나만 생성하여 쓰기 동작을 수행합니다
2. DB Write 전용 스레드, DB Read 전용 스레드를 동시에 진행 후 Write 작업 시간과 Read 작업 시간을 기록하여 Reader 부하가 Writer 성능에 미치는 영향을 비교합니다
=> 관측된 성능 차이는 주로 CPU 점유 및 스레드 스케줄링 경쟁에서 오는 편차이며, WAL 동시성 자체에서 오는 추가적인 성능 저하는 관측되지 않았습니다.'

Reader 스레드가 병행 실행되면서 WAL checkpoint가 지연되었고,
그로 인해 WAL 파일 크기 및 wal-index(.shm) 관리 비용이 증가하여 Write 트랜잭션 성능 저하가 발생했습니다.

3. AddSIM의 배치, 몬테카를로는 반복 수행 횟수만큼 DB를 생성하기에 이때 WAL 모드를 사용했을 때 파일 생성(.db, .wal. shm) 초기 생성 오버헤드를 측정하기 위한 테스트입니다.
